# WP1 — Control Plane → Execution Loop

Goal

Turn Notion from static schema into an active queue: changes produce jobs, jobs can be consumed.

Scope (strict)
	-	Notion webhook receiver
	-	Event → job translation
	-	Idempotent job creation
	-	Minimal job lifecycle (queued → running → succeeded/failed)

Inputs
	-	Existing Notion control plane (data sources + schema)
	-	Integration token
	-	Public endpoint (e.g. small Node/Python service)
	-	Feature flags (must include FEATURE_JOB_CONTROL=1)

Key Activities
	-	Register webhook subscription (page + data source events)
	-	Implement signature verification
	-	Map events → job creation rules:
	-	new/updated Source → ingest_source
	-	Trigger Regeneration=true → update_wiki
	-	Deduplicate via idempotency key:
	-	e.g. source_id + event_type + content_version
	-	Write job rows into Jobs data source

Deliverables
	-	Running webhook service
	-	Job creation logic (stateless)
	-	Idempotency mechanism
	-	Log of received events and created jobs

Acceptance Criteria
	-	Creating or editing a Sources row produces exactly one Jobs row
	-	Duplicate edits do not create duplicate jobs
	-	Job rows contain:
	-	Job Type
	-	Target Source
	-	Job Status=queued
	-	Idempotency Key
	-	Manual toggle of Trigger Regeneration produces a job within <2 seconds

# WP2 — Worker + Source Processing Pipeline

Goal

Consume jobs and turn raw sources into structured intermediate artefacts.

Scope (strict)
	-	Job polling loop
	-	Source fetching + parsing
	-	Intermediate artefact writing
	-	Status updates back to Notion

No wiki generation yet.

Inputs
	-	Jobs data source
	-	Sources data source
	-	Webhook-produced jobs
	-	Storage location (local FS or object store)

Key Activities
	-	Poll Jobs:
	-	filter Job Status=queued
	-	lock job (Locked=true)
	-	Transition:
	-	queued → running → succeeded/failed
	-	Implement source handlers (minimum viable):
	-	web page (URL fetch + clean text)
	-	notion_page (read blocks)
	-	Write artefacts:
	-	raw text
	-	normalised markdown
	-	Store pointers back into:
	-	Raw Text Pointer
	-	Normalised Markdown Pointer
	-	Update:
	-	Source Status=processed
	-	timestamps (Last Parsed At, etc.)

Deliverables
	-	Worker service (single process acceptable)
	-	Source parsing module
	-	Artefact storage (filesystem path or bucket)
	-	Job lifecycle handling

Acceptance Criteria
	-	A queued ingest_source job:
	-	transitions through running → succeeded
	-	Source row updated with:
	-	Source Status=processed
	-	valid pointers (non-empty URLs/paths)
	-	Failures:
	-	populate Parse Error and Error Message
	-	job marked failed
	-	Re-running the same job does not duplicate artefacts (idempotent)

**Implementation note (llmwiki-runtime):** Ingestion sets source status to **parsed** and enqueues a follow-up **`update_wiki`** job; **processed** is set after a successful wiki update. The on-disk layout is **scoped** (`raw/shared/…`, `wiki/shared/…`); see `docs/shared.md`.

# WP3 — Wiki Maintainer (LLM Loop)

Goal

Implement the actual LLMWiki behaviour: sources update persistent markdown pages.

Scope (strict)
	-	Maintainer instruction file
	-	Wiki page generation/update logic
	-	File-based wiki store
	-	Minimal linking + citation behaviour

Inputs
	-	Parsed sources (from WP2)
	-	Policies data source (if enabled)
	-	Wiki output directory (e.g. /wiki/*.md)
	-	LLM API

Key Activities
	-	Define maintainer prompt:
	-	how to update pages
	-	citation format
	-	conflict handling rule
	-	For each job:
	-	load source markdown
	-	identify affected pages (simple heuristic initially)
	-	create/update .md files
	-	Persist:
	-	one file per wiki page
	-	Update Notion:
	-	Wiki Pages rows
	-	relations (Backing Sources)
	-	Last Generated At
	-	Handle:
	-	new page creation
	-	page update (overwrite or diff)

Deliverables
	-	Maintainer prompt (versioned)
	-	Wiki directory with generated .md files
	-	Worker extension for update_wiki jobs
	-	Notion sync (wiki rows + relations)

Acceptance Criteria
	-	One processed source results in:
	-	≥1 markdown wiki file created or updated
	-	Wiki file contains:
	-	structured content
	-	explicit citations to source
	-	Re-running with same source:
	-	produces stable output (no uncontrolled drift)
	-	Updating a source:
	-	updates existing wiki pages (not duplicates)
	-	Notion reflects:
	-	correct Backing Sources relation
	-	updated timestamps
